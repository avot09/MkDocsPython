## Исследование по теме развёртывания

### 1. Использование российских CDN для ускорения загрузки контента

**Сеть доставки контента (CDN)** — это распределённая сеть серверов, предназначенная для повышения скорости и надёжности доставки веб-контента. Принцип работы CDN основан на геораспределении: файлы сайта (изображения, стили, скрипты) кэшируются на множестве серверов, расположенных в разных регионах, и запрашиваются пользователем с ближайшего из них. Это значительно снижает задержки и повышает доступность ресурса.

Российский рынок предлагает ряд решений для эффективного использования CDN:

*   **[Yandex Cloud CDN](https://yandex.cloud/ru/services/cdn)** — интеграция с облачной платформой Яндекса, защита от DDoS-атак.
*   **[VK Cloud CDN](https://cloud.vk.com/docs/ru/networks/cdn)** (ранее — Mail.ru Cloud Solutions) — сервис в составе облачной инфраструктуры VK.
*   **[Selectel CDN](https://selectel.ru/services/additional/cdn/)** — решение, ориентированное на надёжность и производительность.
*   **[CDNvideo](https://www.cdnvideo.ru/)** и **[NGENIX](https://ngenix.net/ecp/cdn/)** — специализированные платформы для обработки медиаконтента и высоких нагрузок.

### 2. Реализация CI/CD на платформе GitVerse

**GitVerse** — это российская платформа для совместной разработки, предоставляющая полный цикл инструментов для управления проектами, аналогичный GitLab. Её ключевое преимущество — встроенная система непрерывной интеграции и доставки (CI/CD).

Принцип работы CI/CD в GitVerse основан на декларативном описании пайплайнов в YAML-файле (`.gitlab-ci.yml`). В этом файле разработчик задаёт последовательность этапов (например, сборка, тестирование, развёртывание). При каждом событии в репозитории (push, merge request) GitVerse автоматически запускает этот пайплайн, выполняя задания в изолированных средах (раннерах). Это позволяет полностью автоматизировать процесс от коммита кода до его публикации в продакшен, обеспечивая согласованность и контроль качества.

### 3. Варианты деплоя статического сайта в продакшен и необходимые инструменты

Развёртывание статического сайта может быть организовано различными способами, в зависимости от масштаба, бюджета и требований к инфраструктуре.

| Вариант деплоя | Описание | Типичные инструменты | Преимущества | Сложность |
| :--- | :--- | :--- | :--- | :--- |
| **Git-ориентированные платформы** | Публикация сайта напрямую из репозитория. | GitHub Pages, GitLab Pages, Netlify, Vercel. | Максимальная простота, полная автоматизация, часто бесплатный старт. | Низкая |
| **Объектное хранилище + CDN** | Загрузка файлов в облачное хранилище с подключением CDN. | Яндекс Object Storage, AWS S3, `s3cmd`, CLI облачных провайдеров. | Высокая производительность, масштабируемость, надёжность. | Средняя |
| **Виртуальный сервер (VPS)** | Размещение файлов на выделенном сервере. | Nginx/Apache, `rsync`/`scp`, Ansible для автоматизации. | Полный контроль над окружением и конфигурацией. | Высокая |
| **Контейнеризация** | Упаковка сайта в Docker-контейнер. | Docker, Dockerfile, Docker Hub, Kubernetes. | Стандартизация, лёгкость переноса между средами. | Высокая |

**Наиболее простым** для старта является использование встроенных возможностей Git-платформ (например, GitHub Pages с GitHub Actions). Для проектов, где критичны **скорость и глобальная доступность**, оптимальным выбором становится связка объектного хранилища и CDN. **Полный контроль** над инфраструктурой обеспечивает развёртывание на VPS, а **контейнеризация** актуальна для сложных сред, где статический сайт — часть более крупной микросервисной архитектуры.
